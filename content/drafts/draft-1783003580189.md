<p>𝗔𝗴𝗲𝗻𝘁𝗶𝗰 𝗔𝗜 𝘀𝗼𝗹𝘂𝘁𝗶𝗼𝗻𝘀 𝘃𝘀. 𝘁𝗼𝗼𝗹𝘀 𝗯𝘂𝗶𝗹𝘁 𝗯𝘆 𝗮𝗴𝗲𝗻𝘁𝗶𝗰 𝗔𝗜 𝘀𝗼𝗹𝘂𝘁𝗶𝗼𝗻𝘀<br><br>Today's challange: For a code migration project, do I let AI Agents handle every task directly, or do I use AI agents to build a tool that then handles the migration?<br><br>Both are viable. Factors I am considering - <br><br>𝗙𝗶𝗿𝘀𝘁, 𝘁𝗵𝗲 𝗲𝗻𝗱 𝗴𝗼𝗮𝗹. A migration factory — where you're migrating code on an ongoing basis — favors building a reusable tool. The engineering cost is paid once, and every subsequent run is fast and cheap. A one-time migration favors using agentic AI directly, since building and maintaining a tool for a single use isn't worth the investment.<br><br>𝗦𝗲𝗰𝗼𝗻𝗱, 𝗵𝗼𝘄 𝗺𝘂𝗰𝗵 𝗱𝗲𝘁𝗲𝗿𝗺𝗶𝗻𝗶𝘀𝗺 𝘁𝗵𝗲 𝘁𝗮𝘀𝗸 𝗻𝗲𝗲𝗱𝘀. Code translation benefits from deterministic parsers — the same input should produce the same output every time. Document generation and business rule extraction are less structured and benefit more from agentic reasoning, where judgment matters more than repeatability.<br><br>𝗧𝗵𝗶𝗿𝗱, 𝘁𝗼𝗸𝗲𝗻 𝗲𝗰𝗼𝗻𝗼𝗺𝗶𝗰𝘀 (𝗜𝗺𝗽𝗼𝗿𝘁𝗮𝗻𝘁 𝘁𝗵𝗲𝘀𝗲 𝗱𝗮𝘆𝘀 :𝗣). A tool only calls an LLM when it breaks or needs an upgrade. An agentic pipeline calls one for every task, every run. Over hundreds of migrations, that difference adds up fast.<br><br>I am landing on a hybrid: structured input, like XML source files, goes through a deterministic parser. Where actual knowledge is needed — understanding what's implemented, generating the business rule behind it — we use agentic AI. The tool does the repeatable work; the agents build and refine the tool, and step in where judgment is required.<br><br>Interested to hear how others are approaching this split on their own projects, or where you've run into problems with either approach.</p><img src="https://media.licdn.com/dms/image/v2/D4E22AQE2EC06xaOXfA/feedshare-shrink_160/B4EZ8fjKPGHoAk-/0/1782940739785?e=1784764800&amp;v=beta&amp;t=HrL-k1P5RMK-vZ7mgwXKL4P8erZrup7sfPYT3yUrkMU" alt="View image"><p></p>
